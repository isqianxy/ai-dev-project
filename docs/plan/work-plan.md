# 工作计划（小步迭代 / 可回滚）

## 0. 约束与原则（贯穿全程）

- **小步可回滚**：每个迭代只交付一个最小能力闭环，随时可回退到上一个稳定点。
- **契约先行**：任何对外 API 变更先更新 OpenAPI，再进入实现与测试。
- **可测试优先**：每个迭代都必须产出最小自动化测试与验收清单。
- **安全默认**：涉及“写操作/副作用”的能力，默认走人工确认（Human-in-the-loop）。
- **可观测可追溯**：每个关键路径都要有结构化事件（SSE）与可回放信息。

## 1. 里程碑拆解（从最基本到能力叠加）

### 当前进度（状态同步）

- **已完成**：M1（骨架）/ M2（契约）/ M3（ReAct v0）/ M4（DeepSeek 模型适配）/ M5（工具系统 v1）/ M5+（ToolProvider + Skill 注入 + 基础路由/回退）/ M6a（工具风险元数据与高风险拦截）/ M6b（高风险拦截联动审批）/ M6c（审批通过后单次放行执行）/ M7（短期记忆滑窗 + Redis 可切换存储）
- **下一阶段**：M10（RAG）

### M1：可运行的最小骨架（不含真实 LLM / 不含真实工具）

- **目标**：后端服务可启动，提供最小 REST 接口与最小 SSE 通道；前端可连接并展示事件流（可用 Mock 事件）。
- **产出物**
  - OpenAPI：定义最小接口集合（见 M2 细化）
  - 后端：Spring Boot 项目骨架、统一错误结构、SSE 事件规范
  - 前端：最小页面（任务提交、SSE 输出面板）
  - 测试：接口契约校验 + 冒烟测试
- **验收标准**
  - 可以创建 Session、提交一个“空任务”，SSE 能连续输出模拟事件并正常结束
  - 断线重连（或前端刷新）不导致服务崩溃，错误码结构一致
- **回滚点**：仅包含骨架与契约，功能不依赖外部系统

### M2：API 契约落地（OpenAPI 作为单一事实来源）

- **目标**：将核心资源模型（Session/Run/Event/Approval）固化为 OpenAPI，并建立变更记录。
- **产出物**
  - `docs/api/openapi.yaml`：最小可用契约（先覆盖创建会话、提交任务、订阅 SSE、查询状态、确认审批）
  - `docs/api/changelog.md`：每次接口变更记录（版本号、破坏性变更说明、迁移建议）
- **验收标准**
  - OpenAPI 可通过校验工具（后续在脚本中固化）
  - 所有接口字段含义明确（ID、时间、状态枚举、错误码）
- **回滚点**：任何契约变更可回退到上一版 `openapi.yaml`

### M3：推理引擎 v0（ReAct 循环骨架 + 终止条件）

- **目标**：实现 ReAct 循环的“状态机骨架”，先不接真实 LLM，可用规则/Mock 产生步骤以验证循环、终止与事件流。
- **能力范围**
  - 状态：THINK / ACT / OBSERVE / FINAL
  - 终止：最大步数、时间预算、不可解判定
- **验收标准**
  - 同一任务可稳定产生多步事件，满足终止条件后结束
  - 每个 step 都有 `stepId`、阶段标识与可追溯信息
- **回滚点**：与外部依赖解耦（可在无 LLM 环境运行）

### M4：模型适配层 v1（接入 1 个模型提供方）

- **目标**：抽象统一的模型接口，先接入 1 个提供方（例如 OpenAI 或本地 DeepSeek），支持通过配置切换。
- **验收标准**
  - 模型调用超时、失败可被捕获并转为结构化事件与错误
  - 支持基础重试与退避（最小实现）
- **回滚点**：可切回 Mock 推理以保持系统可运行

### M5：工具系统 v1（声明式注册 + 参数解析 + 结果回传）

- **目标**：完成工具注册与执行闭环：注解扫描、工具清单暴露、JSON→POJO 参数解析、结果/异常回传为 Observation。
- **验收标准**
  - 参数校验失败返回结构化错误 Observation
  - 工具异常不会导致服务崩溃，且可进入下一步决策/终止
- **回滚点**：可禁用工具系统，仅保留纯推理模式

### M5+：Agent v1 增强基石（MCP / Skill 兼容层）

- **目标**：在不破坏现有工具接口前提下，为后续 MCP 与 Skill 做统一抽象，避免重构。
- **能力范围（建议小步）**
  - `ToolProvider` 抽象（本地注解工具 / MCP 远端工具）
  - 统一工具元数据（name/description/schema/riskLevel）
  - `ReasoningEngine` 的工具选择策略与回退策略
  - Skill（Prompt 模板）按场景注入系统提示词
- **验收标准**
  - 本地工具路径保持兼容
  - 新增 MCP provider 时无需改动核心控制器
  - Skill 配置可通过文件切换并影响模型行为
- **回滚点**：Provider 路由可切回本地工具单一路径
- **当前进展**
  - 已完成：引入 `ToolProvider` 接口、`local` provider 适配、`mcp` provider 占位开关（`agent.mcp.enabled`）
  - 已完成：`Skill` 模板注入（`agent.skill.*`）与 DeepSeek + LangChain4j function calling 主链路（模型主导工具调用）
  - 已完成：模型函数调用链路的工具事件透传（`tool.invoked` / `tool.result`）到 SSE
  - 待完成：更丰富的工具选择策略（多工具评分、风险分级、审批联动）

### M6：安全与人工确认 v1（写工具默认需要审批）

- **目标**：引入权限分级与 Approval 流程；写操作默认不直接执行，先发起审批请求。
- **验收标准**
  - 写工具调用会产生 `approval.requested` 事件
  - 用户同意/拒绝后，系统行为与事件一致且可追溯
- **回滚点**：可全局禁用写工具或强制只读模式

### M7：记忆管理 v1（短期滑窗 + Redis 会话持久化）

- **目标**：实现短期上下文滑窗，落地 Redis 存储会话热数据，支持跨重启恢复基本上下文。
- **验收标准**
  - 上下文超限可裁剪且不破坏关键约束（权限/安全/最近 Observation 优先）
  - Redis TTL 生效，且不会误删运行中的会话
- **回滚点**：可退回到内存态会话（仅开发环境）

### M8：可观测性 v1（结构化事件 + 链路关联）

- **目标**：完善事件模型与日志关联，保证 `traceId/sessionId/runId/stepId/toolCallId` 全链路贯穿。
- **验收标准**
  - 前端可按时间线展示推理链路与工具调用
  - 关键错误可定位到具体 step/toolCall
- **回滚点**：事件字段向后兼容（新增字段不破坏旧客户端）

### M9：稳定性与性能 v1（超时/熔断/流式处理）

- **目标**：补齐超时、限流/配额（最小实现）、熔断（最小阈值策略），并对大结果集强制流式处理。
- **验收标准**
  - 模型/工具异常可触发退避与熔断，系统保持可用
  - 大数据处理不出现一次性加载导致 OOM 的路径
- **回滚点**：可通过配置关闭熔断/限流（仅开发环境）

### M10：RAG v0（Elasticsearch：向量化检索闭环）

- **目标**：在不影响核心 Agent 主链路的前提下，新增「知识入库 → 向量化 → **Elasticsearch** 索引 → kNN 检索 → THINK 上下文注入」的最小闭环（方案见 `docs/design/rag-with-elasticsearch.md`）。
- **范围（最小闭口，小步交付）**
  - **M10a**：ES 索引与 Mapping（`dense_vector` + 租户/kb filter 字段）+ 手工/接口写入少量 Chunk 冒烟
  - **M10b**：固定窗口 Chunk + Embedding 适配 + `_bulk` 写入；MySQL 记录文档/任务状态（可简化为单表）
  - **M10c**：推理侧 RAG 调用：查询 Embedding → ES `knn` + **强制 filter** → Top-K 注入 Prompt
  - **M10d**（可选增强）：`content` BM25 + kNN **混合检索**（RRF 或应用层融合）
- **验收标准**
  - 给定固定小知识集，可检索到相关片段并在推理中引用（带来源标识）
  - ES 超时/不可用：**降级为无 RAG**，主链路仍可完成；日志与 SSE 可观测
- **回滚点**：RAG 特性开关关闭；或清空/停用 ES 索引而不影响 Session/Run 核心路径

## 2. 自动化与脚本（随迭代逐步补齐）

- **契约校验**：OpenAPI 校验、破坏性变更检测（从 M2 起）
- **测试入口**：统一脚本执行单测/集成测试（从 M1 起）
- **质量门禁**：PR 必须通过最小冒烟与契约检查（从 M2 起）

## 3. 建议的迭代节奏（最小步幅）

- 每个里程碑拆成多个 PR，每个 PR 只引入一个闭环能力或一个质量门禁
- 每个 PR 必须包含：变更说明、验收清单、回滚方式、影响面评估
