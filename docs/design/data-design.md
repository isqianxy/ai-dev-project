# 数据设计（MySQL + Redis）

## 1. 目标

为会话、任务执行、推理链路、工具调用与审计提供可追溯的数据基础，同时兼顾性能与回放能力。

## 2. 数据域划分（概念层）

- **会话域（Session）**：会话生命周期、上下文摘要、用户/租户标识、权限上下文
- **运行域（Run/Task）**：一次任务执行实例、状态机、开始/结束时间、终止原因
- **事件域（Event）**：推理步骤、工具调用、异常、确认请求等事件流
- **审计域（Audit）**：高风险操作、写工具确认、权限拒绝等安全关键事件

## 3. Redis Key 规划（建议）

- `session:{sessionId}:context`：短期上下文（滑窗）
- `approval:{approvalId}`：确认请求状态（待确认/已同意/已拒绝/超时）
- `run:{runId}:progress`：运行进度与最后事件游标（用于 SSE 断线重连）

## 4. 数据保留策略（建议）

- 会话热数据在 Redis 设置 TTL
- 关键事件与审计数据落库并可配置归档周期
- 支持按会话/任务维度删除（满足隐私与合规）

## 5. Elasticsearch（RAG 知识检索）

- **选型**：Elasticsearch 8.x（建议 8.10+，生产需固定小版本并做好集群规划）
- **用途**：保存知识 **Chunk** 的原文（`text`，供 BM25）、**向量**（`dense_vector`，供 kNN）及过滤字段（租户、知识库 ID 等）；支持纯向量检索与 **混合检索**（全文 + 向量，见 `rag-with-elasticsearch.md`）。
- **边界**
  - **MySQL**：知识库/文档版本、入库任务状态、业务主数据与审计；**不作为**向量检索引擎。
  - **ES**：面向检索的派生数据（可重建）；与 MySQL 通过 `chunk_id` / `document_id` 等关联。
  - **Redis**：不承担大块向量与全文索引；仍可缓存「热门查询 Embedding」等（可选，非必须）。
- **详细方案**：`rag-with-elasticsearch.md`（索引 Mapping、切分、查询、失败降级、安全 filter）。
